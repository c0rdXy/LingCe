# LingCe V0.0.6 Upgrade Plan

## Goal

V0.0.6 focuses on making LingCe configurable instead of hard-coding product text and exam rules in UI code.

The first version should support:

- Custom app name.
- Custom home subtitle.
- Custom default theme.
- Custom exam duration.
- Custom pass score.
- Custom exam question type distribution.
- Custom score per question type.
- Dynamic exam instructions, header text, status groups, scoring, and result text.

Keep the first version focused on one default exam profile. Leave room for multiple profiles later, but do not build that UI yet.

## Current Hard-Coded Areas

- `core/config.py`
  - `APP_NAME`
  - `WINDOW_TITLE`
  - `QUESTION_TYPES`
  - `EXAM_CONFIG`

- `ui/main_window.py`
  - Home title and subtitle.
  - Help text describing fixed exam rules.

- `ui/practice_mode.py`
  - Practice window title.
  - Fixed question type button list and names.

- `ui/exam_mode.py`
  - Exam title.
  - Exam duration: 90 minutes.
  - Total score: 100.
  - Question distribution: 20 single choice, 20 multiple choice, 4 short answer.
  - Scores: single choice 2 points, multiple choice 3 points.
  - Exam instructions.
  - Right-side answer status groups.
  - Result dialog text.
  - Scoring logic.

## Proposed Config Shape

Runtime user settings should be stored in `data/settings.json`.

Do not commit generated runtime `data/settings.json` unless we intentionally decide to include an example file. Prefer committing defaults in Python code.

Suggested shape:

```json
{
  "version": "0.0.6",
  "app": {
    "name": "灵测 LingCe",
    "subtitle": "通用考试练习平台",
    "show_version": true,
    "default_theme": "light"
  },
  "exam": {
    "name": "默认考试",
    "time_limit": 90,
    "pass_score": 60,
    "allow_submit_with_unanswered": true,
    "auto_submit_when_time_up": false,
    "rules": [
      {
        "type": "single",
        "name": "单选题",
        "count": 20,
        "score": 2,
        "auto_score": true
      },
      {
        "type": "multiple",
        "name": "多选题",
        "count": 20,
        "score": 3,
        "auto_score": true
      },
      {
        "type": "short",
        "name": "简答题",
        "count": 4,
        "score": 0,
        "auto_score": false
      }
    ]
  }
}
```

## Implementation Steps

### 1. Add Default Settings

Create `core/default_settings.py`.

Responsibilities:

- Hold the built-in default settings.
- Provide helper functions for deep-copying defaults.
- Optionally provide a merge function that fills missing fields from defaults.

### 2. Add SettingsService

Create `services/settings_service.py`.

Responsibilities:

- Load `data/settings.json`.
- Use defaults when the file does not exist.
- Recover gracefully from invalid JSON.
- Fill missing fields with defaults.
- Save settings when the user edits them.
- Validate settings before saving.

Useful methods:

- `get_settings()`
- `get_app_settings()`
- `get_exam_settings()`
- `get_exam_rules()`
- `get_app_name()`
- `get_window_title(version)`
- `get_question_type_name(type_key)`
- `save_settings(settings)`

### 3. Replace App Text Sources

Update these UI areas to read from `SettingsService`:

- Main window title.
- Home title.
- Home subtitle.
- Help text system name.
- Practice window title.
- Exam window title.

Keep `core/config.py` defaults for compatibility, but runtime display should come from settings.

### 4. Introduce Exam Rule Handling

Add a small internal model or normalized dict structure for exam rules.

Each rule should include:

- `type`
- `name`
- `count`
- `score`
- `auto_score`

Normalize aliases carefully:

- `short`, `essay`, and `fill` can be grouped as subjective/short-answer types.
- `judge` and `judgement` should be handled consistently.

### 5. Make Exam Question Selection Dynamic

Replace the fixed selection logic in `ui/exam_mode.py`.

Current behavior:

- 20 single choice.
- 20 multiple choice.
- 4 short answer.

New behavior:

- Iterate configured exam rules.
- Pick matching questions by rule type.
- Randomly sample up to the configured count.
- If the bank has fewer questions than requested, use all available questions and show a clear warning.

Recommended first-version shortage policy:

- Allow exam start with fewer questions.
- Show one combined warning dialog that lists missing counts.

### 6. Make Exam Status Groups Dynamic

Right-side status groups should be generated from actual selected rule groups.

Example:

- Single choice group title comes from rule name.
- Multiple choice group title comes from rule name.
- Hidden or zero-count rules should not appear.

Keep the question order grouped by rule order.

### 7. Make Exam Instructions Dynamic

Replace the hard-coded instruction block with generated text.

The text should include:

- Exam duration.
- Computed total score.
- Rule-by-rule question composition.
- Auto-scored and manually scored sections.
- Existing operation instructions.

### 8. Make Scoring Dynamic

Replace fixed scoring:

- Single choice: `+2`.
- Multiple choice: `+3`.
- Short answer: no auto score.

New scoring:

- Find the rule for each question.
- If `auto_score` is true and the answer is correct, add `rule.score`.
- If `auto_score` is false, do not auto-score.
- Compute total possible score from configured rules and actual selected counts.
- Store detailed per-rule score stats for the result dialog.

Important: Prefer computed total score from rules rather than a separate editable total score to avoid inconsistent configs.

### 9. Make Result Dialog Dynamic

Replace fixed result detail lines.

Result should show per-rule details:

- Rule name.
- Earned score.
- Possible score.
- Correct count.
- Actual selected count.
- Score per question.
- Manual scoring note when `auto_score` is false.

### 10. Add Settings Window

Add a settings entry from the main window.

First version fields:

- App name.
- Home subtitle.
- Default theme.
- Exam duration.
- Pass score.
- Per question type:
  - Enabled or disabled.
  - Count.
  - Score.
  - Auto-score flag.

Supported built-in types for the first UI:

- Single choice.
- Multiple choice.
- Judge.
- Fill blank.
- Short answer.

Question count `0` should mean disabled.

### 11. Add Validation

Validation rules:

- App name cannot be empty.
- Exam duration must be greater than 0.
- Pass score cannot be negative.
- Question count must be an integer greater than or equal to 0.
- Score must be greater than or equal to 0.
- At least one exam rule must have count greater than 0.
- Auto-score can be disabled for subjective types.
- Unknown types should be ignored or rejected with a clear message.

### 12. Add Tests

Add focused tests for:

- Default settings loading.
- Missing `settings.json` behavior.
- Invalid JSON recovery.
- Missing field merge.
- Settings validation.
- Dynamic exam question selection.
- Shortage handling.
- Dynamic scoring.
- Dynamic instruction text.
- Dynamic result text.

- Run `python -m compileall -q .`.
- Start app with no `data/settings.json`.
- Start app with a custom app name.
- Start exam with default rules.
- Start exam with changed counts and scores.
- Start exam when the question bank has fewer questions than requested.
- Submit exam and verify score calculation.
- Verify dark and light theme text visibility.
- Confirm `data/user_data.json` and generated `data/settings.json` are not accidentally committed.
